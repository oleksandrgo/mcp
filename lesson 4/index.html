Вступ: Transport layer

JSON-RPC, client-server і канал доставки

Вступ

Client-Server Architecture

MCP Client і MCP Server обмінюються інформацією про використання tools. Користувач спілкується з Host природною мовою — Host через Client надсилає структуровані запити.

  • User — запит природною мовою
  • MCP Client / Host — Cursor, Claude Desktop
  • MCP Server — tools, resources, prompts

Вступ

Структура JSON-RPC

Request (client → server): який tool обрано і з якими аргументами. 

{
  "jsonrpc": "2.0",
  "method": "get_weather",
  "params": { "city": "Paris" },
  "id": 1
}
  • method — що викликати
  • params — аргументи tool
  • id — зіставлення відповіді

Вступ

JSON-RPC Response

Response (server → client): результат виконання tool з тим самим id. 

{
  "jsonrpc": "2.0",
  "result": 20,
  "id": 1
}

Transport визначає, як ці повідомлення подорожують — через stdio або HTTP. Формат завжди JSON-RPC.

Вступ

Що таке Transport layer

MCP Protocol визначає що передається. Transport layer — як JSON-RPC доходить між Client і Server.

  • stdio — stdin/stdout, локальний subprocess
  • Streamable HTTP — POST + GET, опційно SSE

Протокол transport-agnostic: lifecycle однаковий — змінюється лише канал доставки.

Локальний stdio

Що таке stdio

  • Client запускає Server як subprocess
  • JSON-RPC з stdin → відповіді в stdout
  • Один JSON на рядок, без вбудованих переносів
  • stderr — лише для логів

Локальний stdio

command і args

Host сам стартує процес сервера при підключенні:

{
  "mcpServers": {
    "my-server": {
      "command": "python",
      "args": ["path/to/server.py"],
      "env": { "API_KEY": "your_key" }
    }
  }
}

Локальний stdio

Один Client — один subprocess

Cursor і Claude Desktop з одним сервером у конфігу → два окремі subprocess, не один спільний процес.

stdio не масштабується на «один сервер — багато клієнтів». Кожне з'єднання Client↔Server — свій екземпляр процесу.

Локальний stdio

Мінімальний код (stdio)

from fastmcp import FastMCP

mcp = FastMCP("Cool MCP Server")

if __name__ == "__main__":
    mcp.run()  # transport="stdio"

Віддалений MCP Server

Streamable HTTP

Streamable HTTP

Як це працює

LOCAL Host / Client Cursor · Claude Desktop

Користувач на локальній машині

Streamable HTTP
remote MCP server Cloud · один процес

Масштабування

×N Клієнти
1 Сервер

Один remote-сервер можуть використовувати тисячі клієнтів — кожен по HTTP, не subprocess

Streamable HTTP

Що таке Streamable HTTP

  • Один endpoint: POST + GET
  • Кожен JSON-RPC — окремий HTTP POST
  • SSE-стрім для server → client notifications
  • Сесії через Mcp-Session-Id

Streamable HTTP

HTTP transport локально

Той самий transport на http://127.0.0.1:8000/mcp — сервер запускається окремо, Client підключається по url.

Зручно для MCP Inspector і коли один процес обслуговує кілька клієнтів на localhost.

Streamable HTTP

Конфігурація: url

{
  "mcpServers": {
    "remote-server": {
      "url": "https://mcp.example.com/mcp"
    },
    "local-http": {
      "url": "http://127.0.0.1:8000/mcp"
    }
  }
}

Client не запускає subprocess — з'єднується з уже працюючим HTTP-сервером.

Streamable HTTP

Мінімальний код (HTTP)

if __name__ == "__main__":
    mcp.run(
        transport="streamable-http",
        host="127.0.0.1",
        port=8000
    )

Proxy спілкується з Host через stdio, з remote — по HTTP з токеном Authorization: Bearer

Аутентифікація

Приклад з Bearer token

{
  "mcpServers": {
    "todos": {
      "url": "https://mcp.example.com/mcp",
      "headers": {
        "Authorization": "Bearer <access_token>"
      }
    }
  }
}

Remote без auth — лише для dev на localhost. У production auth обов'язковий.

Блок 5

Порівняння

stdio vs Streamable HTTP

Порівняння

stdio

  • ✓ Простота — без мережі, портів, TLS
  • ✓ Ідеально для desktop Host
  • ✓ Client керує життєвим циклом процесу
  • ✗ Один subprocess на одне з'єднання
  • ✗ Не підходить для спільного командного сервера

Порівняння

Streamable HTTP

  • ✓ Один процес — багато клієнтів
  • ✓ Деплой у хмару, Docker, Kubernetes
  • ✓ SSE, сесії, resumability
  • ✗ Складніша інфраструктура
  • ✗ Auth обов'язковий у production

Порівняння

Коли що обирати

  • Локальна розробка, desktop-агент → stdio
  • Спільний сервер для команди → Streamable HTTP
  • Хмарний MCP з OAuth → HTTP + auth
  • MCP Inspector на localhost → HTTP на 127.0.0.1

Логіка tools/resources/prompts не змінюється — лише transport layer.

Deprecated HTTP+SSE

Legacy transport

Deprecated

HTTP+SSE → Streamable HTTP

Було (2024-11-05): два endpoint — SSE + POST, подія endpoint при GET.

Зараз (2025-06-18): один endpoint POST+GET, SSE — частина Streamable HTTP, сесії через Mcp-Session-Id.

Нові проєкти — transport="streamable-http". transport="sse" — legacy.

Фінал заняття

Підсумки

На цій лекції ми розібрали transport layer MCP: локальний stdio (subprocess, command/args) і мережевий Streamable HTTP (url, сесії, auth для remote).

При stdio кожне з'єднання Client↔Server — окремий процес; HTTP дозволяє одному серверу обслуговувати багато клієнтів.

У наступній лекції перейдемо до практики: створимо власний локальний stdio MCP-сервер на Python.
