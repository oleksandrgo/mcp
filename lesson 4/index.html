Вступ

Що таке Transport layer

Transport layer — рівень MCP, який відповідає за фізичну доставку JSON-RPC повідомлень між Client і Server. Протокол визначає що передається; transport — як воно доходить від однієї програми до іншої: через який канал, на якій машині, за яким механізмом.

Аналогія: JSON-RPC — текст листа, transport — спосіб доставки. Той самий лист можна передати кур'єром по кімнаті або поштою через інтернет.

  • stdio — локально на одному ПК через stdin/stdout
  • Streamable HTTP — по мережі через HTTP (POST + GET)

Локальний stdio

Що таке stdio

stdio = standard input / standard output. Це не мережевий протокол — стандартні потоки ОС у кожного процесу: stdin (вхід), stdout (вихід), stderr (логи).

Це байтові труби в пам'яті між двома програмами на одному ПК — без TCP, без IP, без HTTP. Усередині просто текст JSON, по одному повідомленню на рядок.

Локальний stdio

command і args

Host сам стартує процес сервера при підключенні:

{
  "mcpServers": {
    "my-server": {
      "command": "python",
      "args": ["path/to/server.py"],
      "env": { "API_KEY": "your_key" }
    }
  }
}

Локальний stdio

Один Client — один subprocess

Host запускає програму як дочірній процес — наче node server.js у терміналі.

Claude Desktop Host
  • spawn("node filesystem-server.js") → Process #1
  • spawn("python git-server.py") → Process #2
  • spawn("node database-server.js") → Process #3

Кожен сервер у конфігу — окремий процес у Task Manager / Activity Monitor

Приклади

  • 1 Host, 5 серверів у конфігу → 5 окремих subprocess
  • Cursor і Claude Desktop → 2 subprocess, той самий server.js Своя пам'ять і stdin/stdout — стан не спільний

Локальний stdio

Чому stdio не масштабується

Один спільний сервер на всю команду через stdio — не працює.

Cursor spawn("node server.js") → Process #1
Claude Desktop spawn("node server.js") → Process #2

Два запуски одного файлу — процеси не знають один про одного

Streamable HTTP

1 сервер слухає порт

100 клієнтів підключаються → працює

stdio

100 клієнтів → 100 процесів

кожен spawn() → неефективно

Обмеження

  • stdio — лише для desktop і локальної роботи Для спільного командного сервера — Streamable HTTP

Масштабування

×N Клієнти
1 Сервер

Host (Client) ↔ remote server у Cloud — один процес обслуговує тисячі клієнтів по HTTP, без subprocess

Streamable HTTP

HTTP transport локально

Той самий transport на http://127.0.0.1:8000/mcp — сервер запускається окремо, Client підключається по url.

Зручно для MCP Inspector і коли один процес обслуговує кілька клієнтів на localhost.

Streamable HTTP

Конфігурація: url

{
  "mcpServers": {
    "remote-server": {
      "url": "https://mcp.example.com/mcp"
    },
    "local-http": {
      "url": "http://127.0.0.1:8000/mcp"
    }
  }
}

Client не запускає subprocess — з'єднується з уже працюючим HTTP-сервером.

Streamable HTTP

Мінімальний код (HTTP)

if __name__ == "__main__":
    mcp.run(
        transport="streamable-http",
        host="127.0.0.1",
        port=8000
    )

Аутентифікація

Приклад з Bearer token

{
  "mcpServers": {
    "todos": {
      "url": "https://mcp.example.com/mcp",
      "headers": {
        "Authorization": "Bearer <access_token>"
      }
    }
  }
}

Remote без auth — лише для dev на localhost. У production auth обов'язковий.

Блок 5

Порівняння

stdio vs Streamable HTTP

Порівняння

stdio

Характеристика
✓ Простота — без мережі, портів, TLS
✓ Ідеально для desktop Host
✓ Client керує життєвим циклом процесу
✗ Один subprocess на одне з'єднання
✗ Не підходить для спільного командного сервера

Порівняння

Streamable HTTP

Характеристика
✓ Один процес — багато клієнтів
✓ Деплой у хмару, Docker, Kubernetes
✓ Сесії через Mcp-Session-Id
✗ Складніша інфраструктура
✗ Auth обов'язковий у production

Порівняння

Коли що обирати

  • Локальна розробка, desktop-агент → stdio
  • Спільний сервер для команди → Streamable HTTP
  • Хмарний MCP з OAuth → HTTP + auth
  • MCP Inspector на localhost → HTTP на 127.0.0.1

Фінал заняття

Підсумки

На цій лекції ми розібрали transport layer MCP: локальний stdio (subprocess, command/args) і мережевий Streamable HTTP (url, сесії, auth для remote).

При stdio кожне з'єднання Client↔Server — окремий процес; HTTP дозволяє одному серверу обслуговувати багато клієнтів.

У наступній лекції перейдемо до практики: створимо власний локальний stdio MCP-сервер на Python.
