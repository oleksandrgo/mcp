Вступ

Від демо до qauto

На попередній лекції — qauto MCP server з базовим tool signup і перевіркою в Inspector.

Сьогодні розширюємо сервер: додаємо signin, add_car і get_cars.

Мета: агент входить у акаунт, додає авто в гараж і читає список машин — природною мовою, без кліків у UI.

Вступ

Що зробимо

Стек: mcp[cli], httpx, pydantic, python-dotenv.

Додамо signin для входу в існуючий акаунт і методи гаражу: add_car, get_cars.

Спільний get_client() тримає cookie sid після signin або signup для всіх наступних tools.

Блок 2

Налаштування середовища

Залежності для qauto MCP server

Середовище

Що ставимо

mcp[cli] httpx python-dotenv pydantic

Середовище

Чому pydantic, якщо є FastMCP

Раніше вистачало type hints: title: str, priority: str = "normal" — FastMCP сам будував JSON-схему. Це працює лише для плоских простих параметрів.

  • Вкладені структури — обʼєкт із кількома полями всередині
  • Строга валідація — наприклад, priority лише "low" | "normal" | "high"
  • Чітка схема відповіді, а не просто str

Нові методи

signin

Вхід у вже існуючий акаунт через POST /auth/signin. Після успіху в get_client() зберігається cookie sid. 

@mcp.tool()
async def signin(email: str, password: str) -> dict:
    """Увійти в qauto і зберегти сесію."""
    payload = SignInInput(
        email=email,
        password=password,
    ).model_dump()
    client = get_client()
    response = await client.post("/auth/signin", json=payload)
    response.raise_for_status()
    return response.json()

Блок 6

get_client і client JSON

Як описувати MCP tools для моделі

Практики

Що бачить модель

Модель не читає тіло функції. Вона бачить лише імʼя, docstring і схему параметрів.

Тому якість tool = якість опису, а не лише робочий код всередині.

Практики

1. Іменування

Добре

@mcp.tool()
async def signup(...): ...

@mcp.tool()
async def get_profile(...): ...

@mcp.tool()
async def create_task(...): ...

# create_* get_* update_*
# delete_* list_* search_*

Погано

@mcp.tool()
async def task_handler(...): ...

@mcp.tool()
async def process(...): ...

# схожі імена плутають модель:
# update_task vs update_tasks
# → update_task + bulk_update_tasks

Практики

2. Docstring

Модель читає docstring замість тіла функції 

@mcp.tool()
async def create_task(
    title: str,
    priority: str = "normal",
) -> str:
    """Створює нову задачу в системі.

    Використовуй, коли користувач просить додати
    або створити задачу/тікет.
    Не використовуй для оновлення — є update_task.

    Args:
        title: Коротка назва (обовʼязково, не порожня).
        priority: "low", "normal" або "high".
            За замовчуванням "normal".

    Returns:
        Рядок з ID і підтвердженням створення.
    """

Практики

Docstring — чекліст

  • Що робить — одне речення на початку
  • Коли використовувати — і коли не використовувати, якщо є схожий tool
  • Формат / обмеження параметрів, якщо тип цього не покриває (наприклад, UUID)
  • Що повертається — за форматом, не за конкретним значенням
  • Пишіть так, ніби пояснюєте tool незнайомій людині, а не собі

Практики

3. Гранулярність

Добре

# Одна фраза = один tool
create_task(...)
update_task(
  task_id,
  title=None,
  priority=None,
)
delete_task(task_id)

# «створи задачу»
# «онови задачу»
# «видали задачу»

Погано

# Надто широко
manage_task(action=...)
# create|update|delete
# → розмита схема

# Надто дрібно
set_task_title
set_task_priority
set_task_status

Практики

4. Типи і Pydantic

from typing import Literal
from pydantic import BaseModel, Field

class CreateTaskRequest(BaseModel):
    title: str = Field(
        ..., min_length=1,
        description="Назва задачі",
    )
    priority: Literal["low", "normal", "high"] = "normal"
    due_date: str | None = Field(
        None,
        description="Дата YYYY-MM-DD",
    )
  • Literal замість голого str
  • Field(description=...) — бачить модель
  • Опційні поля — завжди з дефолтом

Практики

5. Помилки

@mcp.tool()
async def get_task(task_id: str) -> str:
    """Повертає задачу за ID."""
    try:
        data = await client.get(f"/tasks/{task_id}")
        return str(data)
    except httpx.HTTPStatusError as e:
        if e.response.status_code == 404:
            return f"Задачу {task_id} не знайдено."
        return f"Помилка API: {e.response.status_code}"
    except httpx.TimeoutException:
        return "Сервіс не відповідає, спробуйте пізніше."
  • Читабельний текст як результат — не traceback
  • 404 ≠ 500 ≠ timeout — різні повідомлення
  • Без stack trace, внутрішніх URL і ключів

Практики

6. Побічні ефекти

@mcp.tool()
async def delete_task(task_id: str) -> str:
    """
    Безповоротно видаляє задачу за ID.
    Цю дію не можна скасувати.
    Використовуй лише якщо користувач
    явно підтвердив видалення.
    """
  • Незворотні дії — явно в docstring
  • За потреби патерн draft → confirm
  • Видалення, повідомлення, фінанси — з обережністю

7. Обсяг відповіді

@mcp.tool()
async def list_tasks(
    limit: int = 20,
    status: str | None = None,
) -> str:
    """Список задач (макс. limit, дефолт 20)."""
    data = await client.get(
        "/tasks",
        params={"limit": limit, "status": status},
    )
    return format_tasks_compact(data)
  • Завжди limit / page з розумним дефолтом
  • Компактний формат: id — title — status
  • Не згодовуйте моделі гігантський сирий JSON

Практики

8. Сумісність

Якщо змінюєте сигнатуру існуючого tool (обовʼязковий параметр, інший формат відповіді) — клієнти зі старою схемою можуть зламатись.

Нові параметри — опційні з дефолтом

Додавайте поля як tags: list[str] | None = None, а не як обовʼязкові. Старі клієнти продовжать працювати без змін.

Не міняти формат відповіді без причини

Host уже «знає» стару схему. Раптова зміна структури повернення ламає очікування моделі й інтеграції.

Breaking change → новий tool *_v2

При серйозних змінах заведіть create_task_v2, а старий create_task залиште для зовнішніх клієнтів.

Практики

9. Тестування

async def test_create_task():
